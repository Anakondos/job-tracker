### Phase 2: Auto JD Fetch (Medium effort)  
- [ ] **Fetch JD in daemon** ‚Äî only for NEW + Primary/Adjacent + US/Remote jobs
- [ ] **Cache JD in pipeline.json** ‚Äî avoid re-fetching
- [ ] **Parse deadline from JD text** ‚Äî regex/AI extraction ("apply by", "deadline", etc.)

### Phase 3: Quick Matching (Medium effort)
- [ ] **Keyword matching** ‚Äî JD keywords vs profile keywords (no AI)
- [ ] **Title matching score** ‚Äî how close to target roles
- [ ] **Location score** ‚Äî NC > Neighbor > Remote > Other
- [ ] **Combined relevance score** ‚Äî weighted sum of above
- [ ] **Auto-filter by score** ‚Äî show "Recommended" badge for score > 80%

### Phase 4: AI Matching (High effort, on-demand)
- [ ] **AI matching only for Recommended** ‚Äî limit API calls
- [ ] **Skills gap analysis** ‚Äî what's missing vs JD requirements  
- [ ] **Experience alignment** ‚Äî years/level match
- [ ] **Store AI analysis** ‚Äî cache in job metadata
- [ ] **Matching explanation** ‚Äî why this job is/isn't a good fit

### Phase 5: Auto-Prepare Documents (High effort, on-demand)
- [ ] **Auto-prepare for high-match jobs** ‚Äî score > 90%
- [ ] **Queue system** ‚Äî prepare in background
- [ ] **Notification** ‚Äî "3 jobs ready to apply"
- [ ] **One-click apply** ‚Äî open job + docs ready

### Optimization Rules
- Only process: NEW status + Primary/Adjacent roles + US/NC/Remote
- Rate limit: max 50 JD fetches per daemon cycle
- AI calls: only for "Selected" or "Recommended" jobs
- Cache everything: JD, analysis, matching scores
